I asked Gemini "what linux software cannot run on mainframe linux"

While Mainframe Linux (running on IBM Z and LinuxONE systems) is a full-fledged Linux distribution, certain types of Linux software either cannot run or face significant challenges. The primary reason is the fundamental difference in the underlying CPU architecture (z/Architecture, or s390x) compared to the prevalent x86 architecture.

Here's a breakdown of what generally cannot run or is highly challenging:

  1. x86 Binary-Only Software (Native Compiled Applications):

    • This is the biggest category. Any application compiled specifically for x86 (e.g., .deb or .rpm packages built for amd64 or i386) will not run directly on a mainframe. The CPU simply doesn't understand the x86 instruction set.

    • This includes most commercial, closed-source software that doesn't provide an s390x build. Examples might include:

      • Proprietary CAD/CAM software

      • Specialized engineering or scientific applications

      • Certain gaming engines or games (though mainframes are not gaming machines)

      • Niche security software

    • Solution: You would need the software vendor to provide an s390x-compiled version, or if it's open source, you'd need to recompile it from source code for s390x.

  1. Software Relying on x86-Specific CPU Features/Intrinsics:

    • Applications that heavily leverage x86-specific instruction set extensions like:

      • AVX, AVX2, AVX-512 (as we discussed)

      • SSE, SSE2, SSE3, SSE4

      • FMA, BMI, AES-NI (unless an s390x equivalent is used or emulation is provided)

      • Intel SGX (Software Guard Extensions) or other unique hardware security features.

    • While RISC-V (and by extension, s390x for certain operations) has its own vector extensions and cryptographic accelerators, the code built to use x86 intrinsics directly cannot run. Recompilation would be needed, and the compiler/programmer would have to map these to s390x equivalents or generic code.

  1. Drivers for x86-Specific Hardware:

    • Linux requires drivers to interact with hardware.1 Mainframes have their own unique I/O architecture (channels, channel paths, control units, etc.), which is fundamentally different from the PCI Express-based I/O of x86 servers.2

    • Therefore, any Linux driver written specifically for x86 hardware (e.g., specific NVIDIA GPUs, exotic RAID controllers, certain network cards, USB devices) cannot run on a mainframe.

    • The Linux kernel for s390x includes drivers for mainframe-specific hardware.3

  1. Software that Bundles x86 Binaries:

    • Some software packages, even if written in an interpreted language (like Python), might include compiled binary components or native libraries (.so files) that are pre-compiled for x86.

    • If these native components don't have an s390x version, the software will not function. You'd need to rebuild those components for s390x.

  1. GUI Applications (with caveats):

    • While you can run graphical desktop environments and GUI applications on Mainframe Linux, it's not their primary use case. Mainframes are typically run "headless" for server workloads.

    • Performance for demanding graphical applications (e.g., video editing, 3D rendering, complex IDEs like a full Visual Studio Code desktop client) might not be ideal, as the graphical output would need to be rendered on the mainframe and then displayed remotely (e.g., via VNC or X11 forwarding), incurring latency. The mainframe's strengths are I/O, transaction processing, and massive scalability, not interactive desktop graphics.

  1. Certain Low-Level System Utilities/Bootloaders:

    • Bootloaders like GRUB or LILO (though some s390x equivalents exist, like zipl) are highly architecture-specific and cannot be swapped between x86 and s390x.

    • Low-level debugging tools or kernel modules that directly manipulate x86 CPU registers or specific hardware features would obviously not work.

What can run (and often runs very well):

  • Interpreted Languages: Python, Java (with s390x JVM), Node.js, Ruby, Perl, PHP.4

  • Container Runtimes: Docker, Podman, Kubernetes (with s390x-compatible images).5

  • Databases: PostgreSQL, MySQL, MariaDB, Db2, Oracle (if s390x versions are provided).

  • Web Servers/Application Servers: Apache HTTP Server, Nginx, JBoss, Tomcat.

  • Middleware: Kafka, RabbitMQ, Redis, OpenShift.

  • Cloud Native Tools: Many Cloud Native Computing Foundation (CNCF) projects have s390x support.6

  • Standard Linux Utilities: bash, grep, awk, sed, coreutils, etc.

  • Compilers: GCC, Clang (for s390x).7


In essence, any Linux software that is open-source and well-maintained, adheres to POSIX standards, and doesn't have hardcoded x86-specific binaries or hardware dependencies, has a very good chance of being compiled and running successfully on Mainframe Linux. The IBM Z and LinuxONE platforms are well-supported by major Linux distributions, ensuring a robust open-source ecosystem.8
